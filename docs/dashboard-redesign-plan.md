# Dashboard Redesign — Операционный дашборд

Дата сессии: 2026-05-30

## Суть

Текущий дашборд — копия интерфейса Poster, информирует но не помогает менеджеру делать ежедневную работу. Новый дашборд строится по принципу:

> **Дашборд не информирует — он направляет.** Когда всё нормально — тихо. Когда проблема — она не может остаться незамеченной.

## Пользователь и сценарий

**Пользователь:** менеджер смены (но владелец тоже использует для аналитики).

**Ежедневный цикл менеджера:**
- Утро: открыть смену, проверить сотрудников, зависшие заказы, склад
- День: подтвердить принятые кассиром поставки, проверить расходы и списания, следить за зависшими заказами
- Вечер: закрыть смену, сверить кассу, посмотреть итоги

## Контекст: POS + Админка

- **POS (Expo-приложение):** кассиры и официанты — чеки, заказы, обслуживание. Также могут: создать/принять поставку, внести расход, списать.
- **Админка (этот проект):** менеджер — подтверждает, дозаполняет, проверяет.
- **Оба приложения** шарят одну Supabase-базу.

**Ключевая болячка:** кассир в POS принял поставку и отдал наличные, но менеджер в админке об этом не знает → звонит кассиру «сколько в кассе?». Решение: POS передаёт движение сразу, админка показывает его в реальном времени.

## Решения, принятые в сессии

### Что НЕ включаем в дашборд
- ❌ График выручки на весь экран → заменили на KPI-карточки
- ❌ Методы оплаты (карта/нал/QR) → это финансовый отчёт, не операционка
- ❌ Топ блюд сегодня → заменили на «склад под угрозой» с привязкой к блюдам
- ❌ Три цветовых индикатора (зелёный/жёлтый/красный) → только алерты при проблемах
- ❌ Отдельный блок «Текущая смена» → алерт при >12ч + смена видна в хронологии
- ❌ Heatmap / график загруженности → KPI достаточно

### Касса в KPI
- Показывает: `стартовый остаток + наличные оплаты - наличные расходы - инкассации`
- Формула работает только если POS передаёт ВСЕ движения кассы
- Для моков: заглушка «12 400 сом»
- Алерт по кассе: только если расхождение при ЗАКРЫТИИ смены > 500 сом

### Склад: жёлтый vs красный
- Жёлтый: ингредиента < порога (остаток мал), но точный расчёт дней — только после Фазы А (consumption из `inventory-mvp-plan.md`)
- Красный: ингредиент на исходе, < 1 дня
- Пока (без consumption): просто остаток < N единиц
- Каждая позиция — с привязкой к блюдам: «Молоко → Латте (47), Капучино (31)»

### Графики: ECharts, не Recharts
- Проект уже использует ECharts (`echarts-for-react`). Recharts не добавляем.
- Существующие компоненты (`MonthlyRevenueChart`, `TopItems`, `HourlyHeatmap`) не трогаем — позже можно удалить как dead code.

### Sticky bar с кнопками — да, вверху страницы
- Всегда виден при скролле
- Кнопки ведут на страницы: +Расход → /transactions, +Списание → /warehouse/write-offs, +Поставка → /warehouse/deliveries, Закрыть смену → /cash-shifts

## Фазы реализации

### Фаза 1 — Сейчас (моки)
- Новый файл `src/pages/Dashboard.tsx` (не трогаем старый, создаём новый)
- Все компоненты в `src/components/dashboard/`
- Мок-данные в `src/mocks/dashboardMocks.ts`
- Типы в `src/types/dashboard.ts`
- Подключаем к роуту `/` в `App.tsx`

### Фаза 2 — Реальные данные (потом)
- Замена импорта моков на хуки Supabase
- Для кассы: формула из движений (требует чтобы POS писал все движения)
- Для склада: consumption из order_items × recipe_items (Фаза А из inventory-mvp-plan.md)
- Для хронологии: агрегация из shifts, cash_transactions, warehouse_documents
- Для алертов: реальные запросы (зависшие заказы, поставки без оформления)

- Весь текст на русском
- Tailwind + shadcn CSS-переменные для цветов

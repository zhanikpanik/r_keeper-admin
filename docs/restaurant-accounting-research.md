## 1. План счетов для ресторана (Chart of Accounts)

Правильно построенный план счетов — фундамент. Общий план счетов не ловит
ресторанную специфику: стоимость продуктов и труда размазана по общим статьям,
потеряна видимость главного показателя — **Prime Cost**.

### Категории счетов

| Группа | Что входит |
|--------|-----------|
| **Выручка (Revenue)** | Еда, напитки, кейтеринг, доставка — раздельно |
| **Себестоимость (CoGS)** | Еда, напитки, расходники (салфетки, фильтры), алкоголь — раздельно |
| **Труд (Labor)** | Зал, кухня, управление, налоги на ФОТ, бенефиты |
| **Операционные расходы** | Ремонт, маркетинг, коммуналка |
| **Аренда (Occupancy)** | Аренда, налог на имущество, страховка |
| **Административные** | Бухгалтерия, софт, обучение |

### Российский план счетов (общепит)

Базовые счета для автоматизации:

| Счёт | Назначение |
|------|-----------|
| **10** | Сырьё и материалы (продукты для кухни) |
| **20** | Основное производство (прямые затраты на блюда) |
| **21** | Полуфабрикаты собственного производства |
| **25** | Общепроизводственные расходы |
| **26** | Общехозяйственные расходы |
| **41** | Товары (бутилированная вода, сигареты — что продаётся без обработки) |
| **44** | Расходы на продажу (оформление зала, реклама, амортизация оборудования зала) |
| **50** | Касса (наличные) |
| **51** | Расчётный счёт |
| **57** | Переводы в пути (эквайринг — деньги ещё не дошли до р/с) |
| **60** | Расчёты с поставщиками |
| **62** | Расчёты с покупателями (корпоративы, мероприятия) |
| **90** | Продажи (выручка и себестоимость продаж) |
| **94** | Недостачи и потери |

## 2. Основные проводки

### Поступление товаров/сырья

```
Дт 41 (10)  Кт 60    — товар (сырьё) от поставщика
Дт 41 (10)  Кт 71    — закупка подотчётным лицом
Дт 19       Кт 60    — НДС (при ОСНО)
```

Доставка включается в стоимость: `Дт 41(10) Кт 60`.

### Перемещение на кухню/бар

Обязательно документальное оформление (заборные листы или ПО):

```
Дт 20  Кт 10    — сырьё передано в производство
```

Если сначала делаются полуфабрикаты:

```
Дт 21  Кт 10    — сырьё передано для изготовления ПФ
Дт 20  Кт 21    — ПФ передан в производство блюд
```

### Выручка

```
Дт 50       Кт 90.1   — наличные
Дт 57       Кт 90.1   — эквайринг (терминал)
Дт 51       Кт 57     — эквайринг поступил на р/с (за вычетом комиссии)
Дт 91       Кт 57     — комиссия банка
Дт 62       Кт 90.1   — корпоративы (безнал)
Дт 90.2     Кт 20(41) — списана себестоимость проданных блюд/товаров
```

### Списание

По итогам инвентаризации:

```
Дт 94  Кт 10(41)   — выявлена недостача/порча
Дт 73  Кт 94       — отнесено на виновных
Дт 91  Кт 94       — в пределах норм естественной убыли (уменьшает налоговую базу)
Дт 91  Кт 94       — сверх норм или без виновных (НЕ уменьшает базу)
```

### Закрытие косвенных расходов

Возможен вариант: прямые на 20 счете, косвенные на 44:

```
Дт 90.2  Кт 44     — косвенные расходы списаны на себестоимость продаж
```

## 3. Ключевые KPI ресторана

### 3.1 Food Cost (фудкост)

```
Food Cost % = (Стоимость ингредиентов / Выручка от еды) × 100
```

**Норма:** 28–35%. Всё что выше 35% — повод для разбора.

### 3.2 Labor Cost

```
Labor Cost % = (Общие затраты на труд / Общая выручка) × 100
```

Включает: ЗП, налоги, бенефиты, страховку.

### 3.3 Prime Cost (главный показатель)

```
Prime Cost = CoGS + Total Labor
Prime Cost % = (Prime Cost / Общая выручка) × 100
```

**Норма по типам заведений:**

| Тип | Prime Cost % |
|-----|-------------|
| Quick service (фастфуд) | 55–60% |
| Full service (ресторан) | 60–65% |
| Устойчивый бизнес | ~60% |

Prime Cost = два крупнейших контролируемых расхода: еда + люди.
Остальное (аренда, коммуналка) фиксировано — ими сложно управлять операционно.

**Расчёт CoGS (основа для Prime Cost):**

```
CoGS = Начальные остатки + Закупки − Конечные остатки
```

⚠️ Без точного учёта остатков CoGS считается неверно — и весь Prime Cost тоже.

### 3.4 Оборачиваемость запасов (Inventory Turnover)

```
Turnover = CoGS / Средняя стоимость запасов
         = Себестоимость продаж / Средний остаток на складе
```

**Норма:** 4–8 оборотов в месяц.
- Ниже 4 — избыток, деньги заморожены в продуктах
- Выше 8 — риск сток-аутов

**Пример:** $24,000 себестоимости в месяц → средний остаток должен быть $3,000–$6,000.

### 3.5 Средний чек

```
Average Check = Выручка / Кол-во гостей
```

### 3.6 Оборачиваемость столов

```
Table Turnover = Кол-во гостей / Кол-во столов
```

## 4. Складской учёт: термины и методы

### 4.1 Ключевые термины

| Термин | Определение |
|--------|------------|
| **Sitting inventory** | Остаток продукта на руках (в единицах или деньгах) |
| **Depletion** | Израсходовано за период (в деньгах) |
| **Usage** | На сколько дней хватит остатка = Sitting Inventory / Depletion rate |
| **Shrinkage** | Потери (кража, порча, ошибки) = Потеря ÷ Общий запас × 100% |
| **Variance** | Разница между фактическим расходом и продажами по POS. Например: израсходовано 200 кг, продано 190 кг → variance = 10 кг (5%) |
| **Yield** | Коэффициент выхода = Продажи по POS / Фактический расход. 190/200 = 95% |
| **PAR Level** | Минимальный запас для автозаказа |

### 4.2 Theoretical vs Actual Inventory

POS считает **теоретический** расход (сколько должно уйти по рецептам из проданных блюд).
Реальность отличается из-за:
- Порчи (спойлаж)
- Ошибок приготовления (перепутал ингредиент, пересолил → выбросил)
- Неправильной порционировки (насыпал больше риса чем в техкарте)
- Краж

**Variance = Actual − Theoretical** — главный сигнал проблем на кухне.

### 4.3 FIFO / FEFO

- **FIFO** (First In, First Out) — что раньше пришло, то раньше уходит
- **FEFO** (First Expiring, First Out) — что раньше истекает, то первым в работу

Шаги FIFO:
1. Маркировать поставки датой получения
2. Новое ставить ЗА старым
3. Обучить персонал брать старое
4. Регулярно проверять сроки

### 4.4 PAR Levels (точка автозаказа)

```
PAR = (Средний дневной расход × Время доставки) + Страховой запас
```

Пример: 30 стейков/день, доставка 3 дня, страховой 40 штук → PAR = (30×3) + 40 = **130 стейков**.

### 4.5 Категории потерь (food waste)

| Категория | Пример |
|-----------|--------|
| **Порча** | Истёк срок, испортилось в холодильнике |
| **Тарелка** | Слишком большая порция, гость не доел |
| **Приготовление** | Обрезки, ошибки кухни, пересол |
| **Перезаказ** | Купили больше чем нужно, не использовали |

USDA: 30–40% всей еды в США выбрасывается. Это $161 млрд в год.

## 5. Инвентаризация: как считать

### Периодичность

- **Минимум раз в месяц** — для всего
- **Раз в 2–3 недели** — для скоропорта (молочка, свежие овощи, мясо)
- **Еженедельно** — для алкоголя и дорогих позиций

### Процесс (6 шагов, по NetSuite)

1. **Создать таблицу:** позиция, единица измерения, текущий подсчёт, цена за единицу, общая стоимость
2. **Сгруппировать позиции:** мясо, сухие, молочка, алкоголь — по зонам хранения
3. **Записать количества** в логичных единицах (кг, литры, упаковки)
4. **Записать последнюю закупочную цену** (актуальная стоимость замещения)
5. **Посчитать общую стоимость:** кол-во × цена
6. **Сверить с PAR-уровнями** — что пора заказывать

### Правила

- Считать в одно и то же время (утро до открытия или ночь после закрытия)
- Те же люди считают (консистентность)
- **Два человека** — точность и защита от краж
- Мобильное приложение а не бумага/Excel — быстрее, меньше ошибок

| **P1** | Waste tracking | Категории потерь при инвентаризации (порча/брак/кража) |

| **P2** | Yield по позициям | Факт / Теория × 100% |

### Как считается фудкост в r_keeper (целевая схема)

```
Продажа (POS) → order_items
    ↓
recipe_items (техкарта) → сколько грамм каждого ингредиента в блюде
    ↓
Суммируем по всем продажам за период = Theoretical usage
    ↓
Инвентаризация даёт Actual usage = Остаток на начало + Поставки − Остаток на конец
    ↓
Variance = Actual − Theoretical
    ↓
Food Cost % = (Actual usage в деньгах / Выручка от еды) × 100
```

## 7. Налоговые режимы (Кыргызстан)

Короткая справка (нужно уточнить у бухгалтера):

- **УСН (упрощёнка):** единый налог с оборота, ставка зависит от вида деятельности
- **Патент:** фиксированная сумма, ограничения по площади и обороту
- **ОСНО (общий режим):** НДС + налог на прибыль, нужно при обороте > порога

Для автоматизации важно:
- Разделять выручку по каналам (нал/безнал/эквайринг) — уже есть в r_keeper
- Хранить закупочные цены для расчёта себестоимости — уже есть `products.cost_price`
- Генерировать акты списания — частично есть (write-offs)

## 8. Что говорят на Hacker News

Несколько живых обсуждений ресторанной экономики, софта и учёта.
Самые ценные инсайты — от людей, которые реально работали в индустрии.

### 8.1 Правило 30/30/30/10 (классическая разбивка)

Из обсуждения [The Classic American Diner](https://news.ycombinator.com/item?id=47901067):

```
30% — food costs
30% — labor costs
30% — overhead
10% — profit margin
```

Пользователь `throwup238`: «Хорошее правило, минус profit margin» — т.е. 10% прибыли
это в лучшем случае. В реальности часто 0–5%.

Вариация от `marak830` в [другом треде](https://news.ycombinator.com/item?id=15444758):
25/25/25/25 (food/staff/facility/profit) — прибыль variable.

### 8.2 Процент фудкоста vs абсолютная прибыль

Очень важный спор в [треде про loss-leaders](https://news.ycombinator.com/item?id=15444758).

Классическое правило: **фудкост не должен превышать 25%.** Но `NaOH` привёл
контрпример, который меняет картину:

| Блюдо | Ингредиенты | Цена | Фудкост % | Прибыль |
|-------|------------|------|-----------|---------|
| Бургер | $2.50 | $10 | 25% | **$7.50** |
| Филе миньон | $10 | $25 | 40% | **$15.00** |

> «Проценты ничего не значат по сравнению с живыми деньгами. Филе даёт
> в два раза больше прибыли при "плохом" фудкосте 40%. Ресторанные люди
> слишком зациклены на проценте фудкоста и упускают этот момент.»

**Вывод для r_keeper:** при анализе блюд показывать не только фудкост-%, 
но и **маржинальную прибыль в деньгах** (contribution margin).

### 8.3 Loss-leaders в ресторане

Из того же треда — как работают убыточные позиции:

- **Салаты — убыточны.** Продают за $5.59, себестоимость $5.85. Но гость
  берёт к салату напиток: продажа $1.79, себестоимость $0.20 → **90% маржа**
- **Основное блюдо** может продаваться почти по себестоимости, но
  картошка фри и газировка делают всю прибыль
- Сети (McDonald's и др.) специально прячут цену отдельного бургера,
  заставляя брать комбо — где фри даёт ~70% маржи, напиток >90%

**Суть:** невозможно продавать всё с наценкой. Рынок не даст.
У всех конкурентов бургер стоит $10 — если ты поставишь $14,
никто не придёт. Прибыль делается на допах.

### 8.4 Почему весь ресторанный софт — говно

Классический тред 2013 года: [Ask HN: Why are POS systems all POSes?](https://news.ycombinator.com/item?id=6525511).
До сих пор актуален.

Причины от `fredsanford` (экс-разработчик POS-вендора):

1. **Вендоры невыносимо жадные.** Проект всё ещё использовал Borland
   tools из ранних 90-х. Попытки перейти на Java/C# провалились —
   не смогли достичь функциональности старой Borland-системы.
   Опытных программистов выживали низкими зарплатами.

2. **Реальная сложность.** POS — не «простой CRUD». Это:
   - Налоги в разных юрисдикциях (ставки, exemptions, льготы) — *безумная* сложность
   - Инвентаризация, barcode, закупки, profit reporting
   - Интеграция с ERP/бухгалтерией/поставщиками
   - У каждой вертикали свои процессы: ресторан ≠ винный магазин ≠ зоомагазин

3. **Покупает менеджер, страдает кассир.** Менеджер слушает sales pitch,
   видит чеклист фич и цену. Usability для кассира за $7.25/час — 
   последнее о чём думают.

4. **Фрагментированные требования.** `belthasar`: «Все хотят разного.
   Тяжело сделать продукт с фичами, которые нужны большинству.»

5. **FAQ от `moron4hire` (автор треда):** владельцы ресторанов — одни
   из самых нежелающих тратить деньги на софт. «Они не занятые, они
   ленивые и жадные.» При этом ненавидят свою POS-систему и мечтают
   чтобы кто-то пришёл и всё настроил за них.

**Вывод для r_keeper:** мы в выигрышной позиции. POS + админка —
один проект, одна команда, нет разрыва покупатель/пользователь.
Это ровно то, чего не хватает рынку.

### 8.5 Double-entry accounting — почему это важно

Тред [The Double Entry Counting Method](https://news.ycombinator.com/item?id=32948495)
(2022, 127 points, 101 комментарий).

Ключевой вопрос от `onos`: «Зачем вообще double-entry, если софт не ошибается
как человек при ручном вводе?»

Ответы:

- **Trial balance = автоматическая проверка целостности.**
  `simonjgreen`: «Суммируешь все счета — две колонки должны совпадать.
  Если совпадают — все проводки как минимум полные.»
  Это не просто «чексумма». Это проверка что `Assets = Liabilities + Equity`.

- **Double-entry — на самом деле N-entry.** Минимальное N=2 (одна кредит,
  один дебет). Но в реальности — split-транзакции на 3+ счетов:
  ```
  Credit $1000 Cash
  Debit  $600  Rent:Personal
  Debit  $400  Rent:Office (tax-deductible)
  ```

- **Prepayments / accruals.** `jimnotgym`: оплатил аренду за 3 месяца вперёд
  в сентябре. В single-entry это просто расход сентября. В double-entry
  деньги идут в prepayments account и списываются частями — видна
  реальная прибыль каждого месяца.

- **Plain Text Accounting.** `jonahbenton`: использовал Beancount 10+ лет.
  Текстовые файлы + git. Простая и надёжная альтернатива QuickBooks.

**Вывод для r_keeper:** если админка начнёт генерировать бухгалтерские
проводки, double-entry — единственный правильный путь. Trial balance
даёт автоматическую проверку что ни одна проводка не потерялась.

### 8.7 Craftplan: микро-пекарня на своём софте

[Show HN: Craftplan](https://news.ycombinator.com/item?id=46847690) (2026, 577 points).

Разработчик построил production management tool для пекарни жены.
**Open source** (GitHub: `puemos/craftplan`). Стек: Elixir + Ash + Phoenix LiveView + PostgreSQL.

Что умеет:
- **Versioned BOMs** (bill of materials) с cost rollups — аналог техкарт с калькуляцией
- **Inventory** с lot traceability, demand forecasting, allergen tracking
- **Production batch planning** — дневной план выпуска
- **Purchasing** иReorder planning
- **Semi-products** — полуфабрикаты (например, джем используется в нескольких продуктах)
- **Labor accounting** — учёт труда

Уроки автора (прямая цитата):
> 1. Data structure is almost everything, then comes business logic
> 2. You must have deep domain knowledge of what you are building
> 3. Iterate fast on the views built on top of the data structure

**Вывод для r_keeper:** Craftplan — почти 1:1 аналог того что мы делаем,
но для пекарни а не ресторана. Стоит изучить их модель данных и BOM-логику.
Особенно: versioned BOMs (техкарты с версионированием!), semi-products
(полуфабрикаты = счёт 21!), cost rollups (автоматический пересчёт себестоимости
при изменении цен ингредиентов).

### 8.8 Даже дорогие рестораны сидят на Excel
